# Sentry integration architecture

We use Sentry for production error tracking across both the Flask backend and the React frontend, under a single Sentry organisation named `woodpecker` with two separate projects: `woodpecker-backend` (Python) and `woodpecker-frontend` (TypeScript). Two projects rather than one keeps alert rules, grouping, and noise separated by platform — Python exceptions and JS errors have incompatible shapes and different on-call concerns.

**Backend:** We use `FlaskIntegration` from `sentry-sdk` for exception capture. The integration hooks into Flask's `got_request_exception` signal, which fires before our `@app.errorhandler(Exception)` is invoked, so there is no interference and no double-capture. We do not call `sentry_sdk.capture_exception()` manually in the error handler — FlaskIntegration owns that. We enrich the Sentry scope with the authenticated user's ID in a `@app.before_request` hook via `sentry_sdk.set_user({"id": session["user_id"]})`. This is the one piece FlaskIntegration cannot do on its own. The global error handler's sole responsibility is returning a structured `{"title": "Something went wrong", "detail": "An unexpected error occurred. Please try again."}` JSON `500` response (see ADR-0016 for the envelope contract). `AppError` subclasses (domain 4xx errors) are passed to `ignore_errors` in `sentry_sdk.init()` so expected validation and conflict errors do not generate Sentry events — only true unhandled exceptions do.

We evaluated owning the capture explicitly (no FlaskIntegration, manual `capture_exception()` in the handler) but rejected it because FlaskIntegration's scope isolation per request — pushing and popping the Sentry scope around each request lifecycle — is non-trivial to replicate correctly in a threaded WSGI server. Getting it wrong causes context from one concurrent request to bleed into another's Sentry event.

**Frontend:** We use `@sentry/react`, configured with source map upload via the Sentry Vite plugin so stack traces resolve to original TypeScript. The root React Error Boundary is `Sentry.ErrorBoundary`. User context is set via `Sentry.setUser({ id: user.id })` when the auth context resolves. Frontend Sentry captures only browser-originated errors: JSON parse failures on otherwise-successful (2xx) responses — a backend contract violation the server never sees — and React render crashes caught by `ErrorBoundary`. API error responses (4xx, 5xx) are intentionally not captured on the frontend; the backend already has them with full stack traces, and a duplicate frontend event would add noise without signal. Sentry is disabled entirely when `VITE_SENTRY_DSN` is unset. The runtime environment is tagged via Vite's `import.meta.env.MODE`.
