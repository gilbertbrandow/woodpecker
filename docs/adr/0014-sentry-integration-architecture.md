# Sentry integration architecture

We use Sentry for production error tracking across both the Flask backend and the React frontend, under a single Sentry organisation named `woodpecker` with two separate projects: `woodpecker-backend` (Python) and `woodpecker-frontend` (TypeScript). Two projects rather than one keeps alert rules, grouping, and noise separated by platform — Python exceptions and JS errors have incompatible shapes and different on-call concerns.

**Backend:** We use `FlaskIntegration` from `sentry-sdk` for exception capture. The integration hooks into Flask's `got_request_exception` signal, which fires before our `@app.errorhandler(Exception)` is invoked, so there is no interference and no double-capture. We do not call `sentry_sdk.capture_exception()` manually in the error handler — FlaskIntegration owns that. We enrich the Sentry scope with the authenticated user's ID in a `@app.before_request` hook via `sentry_sdk.set_user({"id": session["user_id"]})`. This is the one piece FlaskIntegration cannot do on its own. The global error handler's sole responsibility is returning a structured `{"error": "Something went wrong"}` JSON `500` response.

We evaluated owning the capture explicitly (no FlaskIntegration, manual `capture_exception()` in the handler) but rejected it because FlaskIntegration's scope isolation per request — pushing and popping the Sentry scope around each request lifecycle — is non-trivial to replicate correctly in a threaded WSGI server. Getting it wrong causes context from one concurrent request to bleed into another's Sentry event.

**Frontend:** We use `@sentry/react`, configured with source map upload via the Sentry Vite plugin so stack traces resolve to original TypeScript. The root React Error Boundary is `Sentry.ErrorBoundary`. User context is set via `Sentry.setUser({ id: user.id })` when the auth context resolves. Unhandled `500` API responses captured inside the `request()` wrapper via `Sentry.captureException()`.
