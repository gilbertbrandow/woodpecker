# Decoy puzzles store accepted moves as JSONB, not as a `moves` text column

Both `LICHESS_TACTIC` and `SCRAPED_POSITIONAL` store their move sequence as a space-separated `moves: Text` column (`"opponent_move solution_move1 ..."`) which the payload builder splits into `SolveContract.plies`. This works because every ply is a single UCI string. `DECOY` cannot use the same shape: the player ply is a *set* of accepted moves, not a single string, and there is no safe delimiter that is also valid UCI. Instead, `decoy_puzzles` stores two separate columns — `opponent_move: Text` (ply[0]) and `accepted_moves: JSONB` (array of `{uci, cp, drop_cp}` objects, from which ply[1] is built as a `list[str]`). The payload builder for `DECOY` is structurally different from the other two, but the `SolveContract` it produces is handled identically by the engine (set-match on ply[1] per ADR 0001).
